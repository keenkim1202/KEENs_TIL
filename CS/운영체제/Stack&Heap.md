# Stack & Heap

프로그램이 실행될 때 메모리는 크게 **코드(Code) / 데이터(Data) / 힙(Heap) / 스택(Stack)** 영역으로 나뉜다. 그중 동적으로 사용되는 영역이 Stack과 Heap이다.

## Stack
- 함수가 수행 중인 경우에만 함수 내 데이터에 접근할 수 있는 임시 메모리 할당 체계이다.
- 함수가 호출되면 지역 변수, 매개변수, 복귀 주소 등이 **스택 프레임(stack frame)** 단위로 쌓이고, 함수 실행이 끝나면 자동으로 할당/해제된다.
- 할당/해제가 포인터 이동만으로 이루어져 Heap에 비해 상당히 빠르다. (상대적)
- Heap 메모리 영역에 비해 저장 공간이 작다.
- 컴파일 타임에 크기가 정해지며, 프로그래머가 직접 관리할 필요가 없다. (알아서 할당/해제되므로)
- 영역을 초과하면 **스택 오버플로우(Stack Overflow)** 가 발생한다. (예: 종료 조건 없는 재귀 호출)

## Heap
- 자동으로 할당 해제되지 않으며, 동적으로 할당된다. (런타임에 크기 결정)
- 처리 시간(액세스 시간)은 Stack에 비해 상당히 느리다. (상대적)
- 객체를 동적으로 생성하면 Heap 메모리 영역에 생성되고, 그 객체에 대한 참조 정보는 Stack 영역에 저장된다.
- Heap에 저장된 데이터는 수행 중인 모든 스레드에서 접근할 수 있어 Stack 메모리만큼 안전하지 않다. (동기화 필요)
- Stack 메모리 영역에 비해 상당히 크다.
- 프로그래머(또는 GC/ARC)가 잘 관리하지 못하면 메모리 누수(memory leak)가 발생할 수 있다.
- 할당/해제가 반복되면 **메모리 단편화(fragmentation)** 가 생길 수 있다.

## Stack vs Heap
|        | Stack | Heap |
|:---:|---|---|
| 위치     | 연속적 메모리 공간 | 불연속한 메모리 공간 |
| 할당 시점 | 컴파일 타임 | 런타임 |
| 관리 주체 | 시스템(자동) | 프로그래머 / GC·ARC |
| 비용     | 작다 | 크다 |
| 처리속도  | 빠르다 | 느리다 |
| 이슈     | 메모리 부족(Stack Overflow) | 메모리 단편화(fragmentation) / 누수 |
| 크기     | 정해짐(고정) | 유연함(가변) |
| 구조     | 선형적(LIFO) | 계층적 |
| 스레드   | 스레드마다 독립적으로 할당 | 모든 스레드가 공유 |
