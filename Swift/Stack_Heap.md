# Stack & Heap (Swift 관점)

> 운영체제 일반 관점의 정리는 [스택 영역과 힙 영역(Stack & Heap)](../CS/운영체제/Stack&Heap.md) 참고.
> 여기서는 Swift의 값 타입 / 참조 타입이 실제로 어디에 어떻게 할당되는지를 정리한다.

## 어디에 할당되는가

| 분류 | 예시 | 할당 위치 |
| --- | --- | --- |
| 값 타입 (Value Type) | struct, enum, tuple, Int/Double 등 | 기본적으로 Stack |
| 참조 타입 (Reference Type) | class, closure, actor | 인스턴스는 Heap, 참조는 Stack |

- **Stack**: 
	- 함수 호출 단위로 쌓이고 함수가 끝나면 자동으로 해제된다(LIFO). 
	- 포인터만 이동하면 되므로 할당/해제가 매우 빠르고 ARC가 개입하지 않는다.
- **Heap**: 
	- 동적으로 할당되며 ARC가 참조 카운트로 생명주기를 관리한다.
	- 빈 공간을 찾아 할당하고 동기화 처리가 필요해 Stack보다 비용이 크다.

> 스택은 크기가 제한적이다. iOS에서 메인 스레드 스택은 약 1MB, 백그라운드 스레드는 약 512KB 정도다.
> 종료 조건 없는 깊은 재귀나 지나치게 큰 지역 변수를 사용하면 **스택 오버플로우**가 발생한다.

```swift
struct ValueType { var x = 0 }
class  RefType   { var x = 0 }

func test() {
    let a = ValueType()   // a는 Stack에 할당
    let b = RefType()     // b(참조)는 Stack, 실제 인스턴스는 Heap에 할당
}
// 함수 종료 → a는 즉시 해제, b가 가리키던 Heap 인스턴스는 참조 카운트가 0이 될 때 해제
```

## 값 타입이 항상 Stack에 올라가는 것은 아니다

값 타입이라도 다음의 경우에는 Heap에 올라간다.

- **값 타입이 참조 타입을 프로퍼티로 가질 때**: 
	- struct 자체는 Stack이어도, 그 안의 class 인스턴스는 Heap에 있다.
- **클로저가 값 타입을 캡처할 때**: 
	- 캡처된 값은 클로저(참조 타입)와 함께 Heap에 저장된다.
- **프로토콜 타입으로 담을 때 (Existential Container)**: 
	- 값이 컨테이너 크기(보통 3워드)를 넘으면 Heap에 박싱된다.
- **컬렉션의 내부 버퍼**: 
	- `Array`, `String`, `Dictionary`는 struct지만 실제 원소를 담는 버퍼는 Heap에 둔다. 
	- (그래서 Copy-on-Write가 가능)

## Copy-on-Write (CoW)

- 표준 컬렉션(`Array`, `Dictionary`, `Set`, `String` 등)은 값 타입이지만, 복사 시 곧바로 Heap 버퍼를 복사하지 않는다.
- 복사된 변수들은 같은 Heap 버퍼를 공유하다가, 어느 한쪽에서 **수정(쓰기)이 일어나는 순간에만** 실제 버퍼 복사가 발생한다.
- 덕분에 값 타입의 안전성(독립성)을 유지하면서 불필요한 복사 비용을 줄인다.

```swift
var a = [1, 2, 3]
var b = a          // 이 시점에는 버퍼를 공유 (실제 복사 X)
b.append(4)        // 쓰기 발생 → 이때 비로소 버퍼가 복사됨
```

- 직접 만든 struct에는 CoW가 자동 적용되지 않는다. 큰 데이터를 값 타입으로 다룰 때는 `isKnownUniquelyReferenced(_:)`로 내부 class 박스의 유일 참조 여부를 확인해 직접 구현할 수 있다.

## ARC와의 관계

- Heap에 올라간 참조 타입만 ARC(Automatic Reference Counting)의 관리 대상이다.
- 참조 카운트 증감, 순환 참조(Retain Cycle), `weak`/`unowned` 모두 Heap에 존재하는 인스턴스에 대한 이야기다.
- 값 타입은 Stack에서 스코프 종료 시 자동 해제되므로 ARC 부담도, 순환 참조 위험도 없다.

관련: [ARC & GC](../iOS/ARC_vs_GC.md), [Class & Struct](Class_vs_Struct.md), [Storage Modifier (Strong, Weak, Unowned)](../iOS/Storage_Modifier.md)

## 성능 관점

"어디에 할당되는가"는 단순한 위치 문제가 아니라 **할당 비용, ARC 오버헤드, 복사 비용, 동시성 안전성**으로 직결된다.

### 1. 할당 / 해제 비용

- **Stack**:
	- 할당과 해제가 스택 포인터를 위아래로 움직이는 것뿐이라 사실상 비용이 거의 없다.
	- 스코프를 벗어나면 한꺼번에 정리되므로 메모리 단편화도 없다.
- **Heap**:
	- 메모리 관리자가 비어 있는 적절한 블록을 찾아 할당해야 하고, 여러 스레드가 공유하는 영역이라 내부적으로 락/동기화 비용이 든다.
	- 할당/해제가 반복되면 단편화가 생겨 추가 비용이 누적될 수 있다.
- 따라서 자주 생성·소멸되는 작은 데이터는 값 타입(struct)으로 두면 Stack 할당의 이점을 그대로 얻는다.

### 2. ARC 오버헤드

- 참조 타입은 Heap에 있으므로 참조가 추가/해제될 때마다 참조 카운트를 증감해야 한다.
	- 이 카운팅은 멀티스레드 환경에서 원자적(atomic)으로 처리되어야 해서 단순 증감보다 비용이 크다.
	- 객체가 자주 전달·캡처되는 코드에서는 이 retain/release 호출이 누적되어 무시할 수 없는 오버헤드가 된다.
- 값 타입은 ARC 대상이 아니라 이 비용 자체가 없다.
	- 단, 값 타입이라도 내부에 참조 타입 프로퍼티를 가지면 그 프로퍼티에 대한 ARC는 여전히 발생한다.

### 3. 복사 비용

- 값 타입은 전달·할당할 때마다 값 전체가 복사된다.
	- 작은 값(`Int`, 작은 struct 등)은 복사가 저렴하지만, 필드가 많은 큰 struct는 사용하지 않는 부분까지 통째로 복사되어 비용이 커진다.
- 참조 타입은 포인터(보통 8바이트)만 복사하면 되므로 크기와 무관하게 복사가 가볍다.
- 표준 컬렉션은 Copy-on-Write로 이 복사 비용을 회피하지만, 직접 만든 큰 struct에는 자동 적용되지 않는다.
	- 큰 데이터를 값 타입으로 다뤄야 한다면 CoW를 직접 구현하거나, 그 데이터를 class로 감싸 참조로 공유하는 선택을 고려한다.

### 정리 (선택 기준)

- 작고 자주 생성되며 독립적으로 다뤄지는 데이터 → **값 타입**: Stack 할당 + ARC 미개입으로 빠르고 안전하다.
- 크기가 크거나, 식별성·상속·공유 가변 상태가 필요한 데이터 → **참조 타입**: Heap·ARC 비용이 있지만 복사 비용과 공유 의미 측면에서 유리하다.
- 다만 위 차이는 대부분 나노초 단위이므로, 추측이 아니라 **Instruments로 측정한 뒤 hot path에 한해 최적화**하는 것이 바람직하다.
